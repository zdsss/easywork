# Workspace: XiaoBai Easy WorkOrder System

制造业工单系统（MES），由三个独立 Git 仓库组成，前后端完全分离。

**强制要求**：每个功能点开发完成后，必须通过对应的测试用例，测试全部通过后该功能才算开发完成。

## 各项目概览

### easywork（后端）
- **技术栈**：Java 21 / Spring Boot 3.2 / MyBatis-Plus 3.5 / Spring Security 6 / JWT / Flyway 9.x / Bucket4j / Caffeine
- **数据库**：PostgreSQL 15（端口 5432，库名 `workorder_db`）
- **缓存**：Redis（端口 6379，`IdempotencyService` 使用 `@Autowired(required=false)` 允许 Redis 缺失时降级）；生产部署使用 Redis Sentinel（3 哨兵 + 主从）
- **限流**：登录接口 10 次/分钟/IP（HTTP 429）；设备接口 60 次/分钟/IP，由 `RateLimitFilter`（Bucket4j）实现
- **监控**：`/actuator/prometheus` 暴露 Prometheus 指标
- **运行端口**：`8080`
- **Swagger**：`http://localhost:8080/swagger-ui.html`
- **包路径**：`com.xiaobai.workorder`
- **枚举类型**：`WorkOrderStatus`、`WorkOrderType`、`DependencyType`、`OperationStatus` 均为 Java Enum，映射 DB VARCHAR
- **状态机**：`WorkOrderStateMachine` 集中定义所有工单状态流转规则（`canTransition` / `allowedTransitions`）

### easywork-admin（管理端前端）
- **技术栈**：Vue 3 / Vite 5.4.x / Element Plus / Vue Router / Pinia / Axios
- **运行端口**：`5173`
- **对接 API**：`/api/auth/**`、`/api/admin/**`
- **使用角色**：ADMIN（PC 浏览器）

### easywork-worker（工人端前端）
- **技术栈**：Vue 3 / Vite 5.4.x / Vant / Vue Router / Pinia / Axios
- **运行端口**：`5174`
- **对接 API**：`/api/auth/**`、`/api/device/**`
- **使用角色**：WORKER（移动端浏览器，工业 PDA）
- **硬件输入**：`useHardwareInput.js`（扫码枪识别 50ms 阈值、方向键导航、数字快捷键、ESC 返回）；`KeyHints.vue`（固定在 tabbar 上方的快捷键提示条）

## API 权限规则

| 路径前缀 | 角色要求 | 说明 |
|----------|----------|------|
| `/api/auth/**` | 公开 | 登录接口 |
| `/api/device/**` | WORKER 或 ADMIN | 工人端接口 |
| `/api/admin/**` | ADMIN | 管理端接口 |
| `/api/mes/**` | ADMIN | MES 集成接口 |

**API 响应格式：**

```json
{ "code": 200, "message": "Success", "data": { ... }, "timestamp": 1234567890 }
```

- `code === 200`：成功，取 `data`；其他 code：失败，取 `message` 展示错误

**认证机制：** JWT Token 存入 `localStorage`（key: `token`），请求头 `Authorization: Bearer <token>`，有效期 24 小时；JWT secret 必须通过环境变量 `${JWT_SECRET}` 注入（无默认值）。

## 工单状态流转

```
生产工单 (PRODUCTION):
  NOT_STARTED → STARTED → REPORTED → INSPECT_PASSED → COMPLETED
                        ↑            → INSPECT_FAILED → (返工后重新报工)
                        │            → SCRAPPED（报废终态）
                        └──── 撤销报工（STARTED）

检验工单 (INSPECTION):
  NOT_STARTED → STARTED → COMPLETED（报工即完成，不走质检）

转运工单 (TRANSPORT):
  NOT_STARTED → STARTED → COMPLETED

安灯工单 (ANDON):
  NOT_STARTED → STARTED → COMPLETED
```

强制开工配置：`app.workorder.force-start-before-report` 支持按 `orderType` 分别配置（map 结构），默认全部为 false。

**OperationStatus 枚举值：**

| 值 | 说明 |
|----|------|
| `NOT_STARTED` | 未开工 |
| `STARTED` | 已开工 |
| `REPORTED` | 已报工（PRODUCTION 工序） |
| `INSPECTED` | 已检验（INSPECTION 工序） |
| `TRANSPORTED` | 已转运（TRANSPORT 工序） |
| `HANDLED` | 已处理（ANDON 工序） |
| `COMPLETED` | 已完成 |

## 重要注意事项

- 后端使用 `SessionCreationPolicy.STATELESS`，前端不依赖 Cookie/Session
- MES 集成默认关闭（`app.mes.integration.enabled: false`），联调时按需开启
- 逻辑删除字段：`deleted`（1=已删除，0=正常）
- **Node.js 版本**：前端 Vite 5.x 要求 Node.js 18+（LTS）；不要将 vite 升级到 7.x（Node < 20.12 不支持 `crypto.hash`）
- **Flyway**：迁移文件当前版本 V1.9，由 Flyway 自动应用（`baseline-on-migrate: true`，`baseline-version: 1.6`）；集成测试通过 Testcontainers 自动应用所有迁移
- **CONDITIONAL 依赖类型**：已从管理端 UI 移除，`condition_expression` 列保留在 DB 中（历史数据）
- **工序依赖执行**：仅 SERIAL 类型前置工序阻塞开工；PARALLEL 不阻塞
- **Plans 存放位置**：`easyworks\.claude\plans\`（不要写入 `C:\Users` 下）
- **集成测试**：使用 Testcontainers 自动启动 PostgreSQL 容器，无需手动 `docker compose up`；通过 `-Dgroups=integration` 激活，`mvn test` 默认仅运行单元测试

### 测试分层策略

**单元测试（Service 层）**
- 注解：`@ExtendWith(MockitoExtension.class)`
- 所有 Mapper/Repository 依赖使用 `@Mock`；被测 Service 使用 `@InjectMocks`
- 覆盖重点：业务规则、状态机流转、异常分支、边界值
- 不需要：Spring 上下文、数据库、Redis

**Controller 层测试（Web Layer Test）**
- 注解：`@WebMvcTest(XxxController.class)` + `@ActiveProfiles("test")`
- 所有 Service 依赖使用 `@MockBean`
- 覆盖重点：HTTP 状态码、@Valid 参数校验、响应体结构、权限控制（403/401）

**集成测试**
- 注解：`@Tag("integration")`，继承 `PostgresIntegrationTestBase`
- Testcontainers 自动启动 PostgreSQL 16 容器，Flyway 完整迁移
- 执行：`mvn test -Dgroups=integration`

**Security 辅助：**

```java
mockMvc.perform(get("/api/admin/work-orders")
    .with(user("admin").roles("ADMIN")))
    .andExpect(status().isOk());
```

### 测试命名规范

- 格式：`should_[期望结果]_when_[场景条件]`
- 示例：`should_throwException_when_orderNumberDuplicated()`
- 每个测试方法只验证一个行为
- 结构：**Arrange**（准备数据）/ **Act**（执行方法）/ **Assert**（断言结果）
- 异常测试：`assertThatThrownBy(() -> ...).isInstanceOf(BusinessException.class)`
